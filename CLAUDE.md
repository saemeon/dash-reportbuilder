# dash-reportbuilder

Capture, assemble, and export reports from Plotly Dash applications.

Part of the [brand-toolkit](../CLAUDE.md) workspace. Depends on `dash-capture` for the
capture pipeline. Three export targets: Word (`.docx`), PowerPoint (`.pptx`),
Typst source / PDF.

## Architecture (post-refactor, v0.2)

Three pieces:

```
ReportElement  ──render_into──▶  ReportBackend  ──build──▶  bytes
(knows what)                     (knows how to format)
        ▲
        │
   Report (a list of elements)
```

### Core protocols

- [`ReportElement`](src/dash_reportbuilder/protocols.py) — anything with a
  `render_into(backend) -> None` method.
- [`ReportBackend`](src/dash_reportbuilder/protocols.py) — exposes generic
  primitives (`add_image`, `add_heading`, `add_paragraph`, `add_caption`,
  `add_table`, `add_page_break`) plus `build() -> bytes`.

### Built-in elements

In [`src/dash_reportbuilder/elements.py`](src/dash_reportbuilder/elements.py):

- `HeadingElement(text, level=1)`
- `ParagraphElement(text)`
- `ImageElement(data_uri, title=None, caption=None, width_mm=None)`
- `CaptionElement(text)`
- `PageBreakElement()`

Each element is a dataclass with `to_dict()` / `from_dict()` for FileStore
round-trip. Custom elements register via `register_element_type(cls)`.

### Backends

In [`src/dash_reportbuilder/backends/`](src/dash_reportbuilder/backends/):

- `DocxBackend(template=None, title=None)` — python-docx;
  escape hatch: `modify(fn: Callable[[Document], None])`
- `PptxBackend(template=None)` — python-pptx;
  escape hatch: `modify(fn: Callable[[Presentation], None])`
- `TypstBackend(template=None, title=None)` — Typst source + PDF compile via
  `typst` CLI; escape hatch: `append_raw(source: str)`. Also has
  `build_source() -> str` if you want the raw `.typ` text without compiling.
- `HtmlBackend(template=None, title=None)` — self-contained HTML document
  with images embedded as data-URIs;
  escape hatch: `append_raw_html(source: str)`. Also has
  `build_source() -> str` for the document as text.
- `ImageZipBackend()` — collects every image in the report into a `.zip`;
  non-image primitives (headings, paragraphs, etc.) are silent no-ops.
  Filenames default to `image_001.png`, `image_002.png`, …; the `title`
  argument on `add_image()` overrides (sanitized).
  Escape hatch: `add_raw(filename, data)` for arbitrary zip entries.

### Native rendering pattern

When an element wants format-native control:

```python
class FancyTable:
    def render_into(self, backend: ReportBackend) -> None:
        if isinstance(backend, TypstBackend):
            backend.append_raw(self._as_native_typst())
        elif isinstance(backend, HtmlBackend):
            backend.append_raw_html(self._as_native_html())
        elif isinstance(backend, DocxBackend):
            backend.modify(self._build_docx_table)
        else:
            backend.add_table(self.headers, self.rows)
```

No format-specific Protocols; just `isinstance` + the backend's escape hatch.

### Report facade

[`src/dash_reportbuilder/model.py`](src/dash_reportbuilder/model.py) defines
`Report`. It's a list with a couple of helpers:

```python
report = Report(title="Q1")
report.add(HeadingElement("Intro", level=2))
report.add(ImageElement(data_uri=...))
report.export(DocxBackend(template=DocxTemplate(...)))  # → bytes
```

`Report` also supports `remove(item_id)`, `reorder(item_ids)`,
`update_item(item_id, **fields)`, `to_dict()` / `from_dict()` for
serialization.

## Conventions specific to this package

- **`uv.workspace` member**: dash-reportbuilder is registered in the top-level
  workspace `pyproject.toml` and tests run from the workspace `.venv`. Use
  `uv run --active python ...` from anywhere; `uv sync` from repo root if deps
  drift.
- **Optional deps**: `python-docx` and `python-pptx` are installed via the
  workspace test group (top-level `pyproject.toml`). Production users install
  via `pip install dash-reportbuilder[docx,pptx]`.
- **Typst CLI**: required for `TypstBackend.build()` (PDF compile).
  `build_source()` works without it.
- **No `__init__.py`** in test directories; uses pytest's importlib mode.

## Design decisions worth knowing

- **Single `ReportElement` Protocol with `render_into(backend)`.** Considered a
  multi-tier protocol (Document/Tabular/Asset/Mixed); rejected for v1 as
  over-engineered. See § Future Extensions below if Excel/asset modes become
  needed.
- **No format-specific Protocols** (`TypstNative`, `DocxNative`, etc.).
  Elements use `isinstance(backend, TypstBackend)` to opt into native
  rendering. See § Considered Alternatives below for the full reasoning.
- **Per-backend escape-hatch shape varies on purpose.** `append_raw(str)` for
  text-output backends (Typst); `modify(fn)` for object-output backends
  (python-docx, python-pptx). They're different operations; a unified name
  would be misleading.
- **`Report` is dumb on purpose.** Just a list with light reordering helpers.
  All intelligence lives in elements (what to render) and backends (how to
  format).

## Branded templates

The package ships four example template assets under `templates/`:

- `example.typ` — Typst preamble with brand colors, headings, figure/table styles
- `example.html` — `<head>`/`<style>` block with the same brand palette
- `example.docx` — Word base document with branded heading + caption styles, header, page numbers
- `example.pptx` — PowerPoint base with brand bar, title slide, branded title placeholders

Use them via the `example_template_path()` helper:

```python
from dash_reportbuilder import (
    DocxBackend,
    DocxTemplate,
    Report,
    example_template_path,
)

template = DocxTemplate(template_path=str(example_template_path("docx")))
backend = DocxBackend(template=template, title="My Report")
data = Report(title="My Report").export(backend)
```

The example assets are intended as starting points — copy and edit for your
own brand. The Typst file is plain text; the .docx and .pptx are regenerated
from `scripts/build_example_templates.py` (run manually and committed).

## Future extensions (not in v1)

These compose cleanly with the current design — no architectural rewrite needed:

- **Tabular mode** — add an optional `TabularSource` Protocol for elements
  that have underlying data; new `ExcelBackend` queries elements implementing
  it. Rejected for v1 because dash-capture sources are screenshots (no
  underlying data).
- **Template scaffolds** — let document backends accept a "scaffold" template
  (header + element-loop + footer) for narrative reports with fixed prose
  around dynamic elements. Equivalent of knitr/Rmd / `.r.typ`.

## Inspirations

The design was informed by two existing R projects (vendored under
`spitex-benchmark/` and `template-kennzahlenvergleich/` for reference):

- **spitex-benchmark** — Rmarkdown-based report system. Validated the
  separation of styling (`layout.R`) from element generation (`res_*.R`)
  from composition (`.Rmd`). Key takeaway: composition layer should be
  format-independent and elements should be reusable across reports. Their
  global-environment data passing is the anti-pattern we explicitly avoid in
  the typed-element approach.
- **template-kennzahlenvergleich** — Shiny app with **R6 class hierarchy**
  for elements (`R6_EvalElement_BaseClass` with abstract methods like
  `cat_for_typst_report()`, `cat_for_html_report()`, `get_table_for_excel()`).
  This is the closest cousin to what we built. Their model has format-specific
  methods on each element (O(N×M) — every element knows every format); ours
  inverts that: the backend exposes generic primitives, elements call into
  them, and use `isinstance` only for native opt-in. Adding a new backend
  doesn't require touching every element class.

## Considered alternatives

Earlier design rounds explored richer protocols. They were rejected as
over-engineered for v1:

- **Multi-tier protocol** (Document/Tabular/Asset/Mixed). Modeled four export
  paradigms — narrative documents, Excel-style data extraction, image-only
  asset zips, and combinations. Validated by template-kennzahlenvergleich
  (which does all four). Skipped for v1 because dash-reportbuilder's actual
  source — dash-capture screenshots — only supports the Document mode. Listed
  under § Future extensions; can be added without rewriting the core.
- **Format-specific Protocol classes** (`TypstNative`, `DocxNative`,
  `PptxNative`, `HtmlNative`). Earlier sketch had each element implementing
  one Protocol per backend it supported, with backends running
  `hasattr(element, 'render_typst')` checks. Replaced with `isinstance` in
  the element body — same flexibility, simpler module surface, better type
  inference (the type checker narrows the backend type after `isinstance`).
- **Templates as a knitr-style narrative layer**. Considered a Jinja2 template
  with element placeholders so users could write prose interspersed with
  rendered elements. Rejected for v1 because the imperative API
  (`report.add(element)`) covers the dash-reportbuilder use case; templating
  is additive when narrative reports become a real need.
